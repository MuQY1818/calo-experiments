# Repository Guidelines

## Project Structure & Module Organization
- **`sebs/`:** Orchestrates deployments, caching, and provider integrations; extend this package when adding new platforms or orchestration features.
- **`benchmarks/` 与 `benchmarks-data/`:** 包含函数源码及样例输入，维护新基准时保持目录编号命名，例如 `110.dynamic-html`。
- **`config/`, `templates/`, `dockerfiles/`:** 存放示例配置、打包模板和基础镜像定义，提交更改时同步更新相应文档。
- **`tests/`:** 以 `aws/` 子目录和 `test_runner.py` 组织的单元与集成测试，`docs/` 提供深入的使用与架构说明。

## Build, Test, and Development Commands
- **安装依赖：** 运行 `./install.py --aws --azure --gcp --openwhisk --local` 创建 `python-venv`，随后使用 `. python-venv/bin/activate` 激活环境。
- **运行基准：** `./sebs.py benchmark invoke <benchmark> <input> --config config/example.json --deployment <provider> --verbose` 构建并调用单个基准。
- **回归套件：** `./sebs.py benchmark regression test --config config/example.json --deployment <provider>` 执行平台级验证。
- **测试入口：** `python tests/test_runner.py --deployment aws azure` 并行执行选定平台测试；新增平台时追加对应参数。
- **本地调试：** `./sebs.py local start ... --remove-containers` 与 `./sebs.py storage start/stop` 支持 Docker 本地评估。

## Coding Style & Naming Conventions
- **风格基准：** Python 代码遵循 Google Style，使用四空格缩进、类型注解和完整 docstring。
- **命名约定：** 采用 `snake_case` 函数、`CapWords` 类、`UPPER_CASE` 常量；基准目录沿用数字前缀加描述。
- **静态检查：** 使用 `black sebs tests`, `flake8 sebs tests`, `mypy sebs` 保持格式、风格与类型一致性，提交前确保全部通过。

## Testing Guidelines
- **框架：** 测试基于 `unittest` 与 `testtools` 并发执行；在 `tests/<provider>/test_<feature>.py` 中添加新用例。
- **覆盖要求：** 新功能需包含单元测试或回归基准，确保平台行为与性能数据可复现。
- **运行策略：** 修改部署或缓存逻辑时，同时执行 `benchmark regression` 以验证多平台兼容性，并清理 `cache/` 以避免脏数据。

## Commit & Pull Request Guidelines
- **提交信息：** 参考历史记录使用 `[scope] 短描述` 模式，例如 `[dev] Improve regression logging`，必要时追加 issue 引用。
- **PR 要求：** 提供变更摘要、测试结果、配置更新及潜在回滚步骤；涉及云资源时注明所需凭证或最小权限。
- **评审准备：** 在 PR 中链接相关文档章节，标注受影响的基准与平台，确保 reviewer 能复现 `tests/test_runner.py` 与关键命令。
- 记得重大更新后提交代码

## Environment & Configuration Notes
- **虚拟环境：** 始终在仓库生成的 `python-venv` 内安装依赖，避免污染全局 `base` 环境。
- **敏感配置：** `config/*.json` 包含示例凭证占位符，提交前确认未写入真实密钥；部署测试使用 `.gitignore` 保护临时输出。
- **缓存管理：** 变更函数代码或输入后，利用 `--update-code` 与 `--update-storage` 强制刷新缓存，必要时手动清理 `cache/`。

## Paper Writing Notes
- **正文口吻：** 论文正文、图注、表题、摘要、引言、结论都必须按照最终投稿版本来写，不能出现“针对审稿人回应”“Reviewer 提出”“本次修改/修回”“revision/rebuttal”这类措辞。
- **回应位置：** 审稿意见的逐条回应、补实验动机、修订说明只允许写在 `summary_of_changes.tex` 或独立 response 文档中，不要写进主论文正文。
